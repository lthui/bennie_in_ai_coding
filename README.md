# 📑 AI 驱动的低代码编排框架：需求、设计与实施选型报告

## 1. 业务背景与目标
本项目旨在为现有的 **Camunda 编排层**（上层）与 **Commander 执行端**（下层）之间引入一个 **AI 设计师层（AI Harness）**。其核心任务是：
1.  **翻译**：将自然语言需求转化为节点与连线的图谱。
2.  **粘合**：解决自定义节点（如 Ocean, if-else 等）之间的数据流转冲突。
3.  **进化**：通过人工 Review 后的反馈，提升 AI 的编排准确度。

---

## 2. 核心需求分析
### 2.1 语义编排需求
* **Skill 识别**：AI 需理解每个节点的业务内涵，并能根据意图自动挑选。
* **逻辑连线**：不仅是连接节点，更要生成符合 Camunda 规范的分支条件（Gateway Logic）。
* **胶水逻辑（Glue Code）**：自动生成用于数据转换的微型脚本，处理上游输出与下游输入不匹配的问题。

### 2.2 反馈与学习需求（Closed-Loop）
* **差异捕捉（Delta Tracking）**：系统需自动对比“AI 生成的设计”与“人工修正后的设计”。
* **反馈回传**：将修正点（如变量名改写、节点替换、逻辑跳转修正）作为语义反馈喂给 AI。
* **分析学习**：AI 需通过历史修正案例，优化后续生成的参数预测和节点选择策略。

---

## 3. 系统架构设计方案

### 3.1 抽象层：技能化封装（Skill-as-a-Service）
不再向 AI 开放原始的 JSON 协议，而是将其封装为 **Skill**：
* **语义层**：描述 Skill 的业务价值（AI 看的）。
* **协议层**：定义严格的输入/输出（Harness 做的）。
* **实现层**：映射到 Commander 的具体指令（执行端跑的）。

### 3.2 运行机制：黑板模式与变量治理
为了保证流程“丝滑”，AI 必须在生成编排段时：
* **变量自动命名**：遵循一套统一的上下文命名规范（Context Namespace）。
* **隐式衔接**：通过全局上下文（Blackboard）传递数据，减少物理连线的耦合。

---

## 4. 实施选型对比 (Implementation Options)

以下是四种不同的技术实施路径，各自代表了不同的设计哲学：

| 维度 | **Option 1: 模型驱动 (如 PydanticAI)** | **Option 2: 任务流驱动 (如 ControlFlow)** | **Option 3: 全栈代理 (如 Mastra)** | **Option 4: 自研语义桥接 (Custom DSL)** |
| :--- | :--- | :--- | :--- | :--- |
| **设计哲学** | **约束即自由**。通过强类型 Schema 限制 AI 的输出范围，确保绝对合规。 | **过程即设计**。将设计流程拆解为多个子任务，AI 逐步推导。 | **能力即服务**。提供一套完整的 TS 工具链，将 AI 能力深度集成进前端设计器。 | **协议即意图**。完全基于 Prompt 工程映射自定义 DSL。 |
| **优点** | **精度极高**。在处理复杂 JSON 协议时几乎不产生幻觉；非常容易通过对象对比实现“反馈环”。 | **动态性强**。原生支持在设计过程中插入人工确认环节（HITL），边设计边确认。 | **开发体验佳**。与现代前端框架集成丝滑；适合快速构建高交互的 AI 设计器。 | **零框架债务**。不需要学习新框架；完全根据 Camunda 协议量身定制。 |
| **缺点** | 需要对所有自定义节点进行预先的强类型建模（Python 类型提示）。 | 框架逻辑相对较重，设计态的任务管理可能与运行态混淆。 | 生态偏向 TS；在 IDP 等重算法、重 Python 的场景下，后端集成稍显复杂。 | **维护成本极高**。随着节点增多，Prompt 长度爆炸，AI 的稳定性会线性下降。 |
| **反馈进化实现** | **容易**。通过对比模型状态差异，提取结构化 Feedback。 | **中等**。可以记录任务流中的“重试”或“修改”节点作为反馈。 | **极佳**。由于靠近前端，可以非常精细地捕捉用户点击和拖拽轨迹。 | **困难**。很难从非结构化的 Prompt 改动中提炼出稳定的学习逻辑。 |

---

## 5. “反馈进化”模块的详细设计

无论选择哪种框架，反馈闭环的设计应遵循以下逻辑：

1.  **捕捉阶段 (Capture)**：
    * 用户在低代码界面点击“保存”时，系统执行 `compare(AI_Output, Human_Final)`。
    * 记录 **增加、删除、修改** 三类原子操作。
2.  **分析阶段 (Analysis)**：
    * AI 分析修改原因：是“节点选错了”？还是“参数配错了”？亦或是“胶水逻辑写错了”？
3.  **应用阶段 (Learning)**：
    * **短期应用**：将修正后的 JSON 作为 **Few-shot Example** 加入 Prompt，下次生成类似需求时优先级最高。
    * **长期应用**：定期统计高频修正点，反馈给研发人员，用于优化 Skill 的语义描述或调整胶水代码模板。

---

## 6. 结论建议

* 如果你的项目追求**极高的指令准确性**和**清晰的结构化反馈**，**Option 1** 是稳健的选择。
* 如果你希望设计过程本身就是一种**对话式的人机协作**，**Option 2** 更具优势。
* 如果你希望实现**极简的开发链路**且不想维护外部 AI 框架，**Option 4** 适合初期原型。
* 如果你追求**极致的设计端交互体验**且团队擅长 TS 栈，**Option 3** 表现最佳。
